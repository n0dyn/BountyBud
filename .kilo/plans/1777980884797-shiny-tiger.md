# BountyBud MCP Portability & Stability Plan
**Document:** 1777980884797-shiny-tiger.md  
**Date:** 2026-05-05  
**Objective:** Diagnose failing MCP server in multiple IDE / client setups *and* make BountyBud's memory, storage, and learning systems fully portable so that **any directory** you invoke the MCP server from (via any IDE or CLI) continues the same bounty hunt.

---

## Root Cause Analysis – MCP Failures Across Applications

### Evidence Collected
- `mcp_server.py:69`  
  ```python
  DATA_DIR = os.path.expanduser("~/.bountybud")
  ```
  This is **hard-coded global path**.
- All state (targets, primitives, findings, sessions, hunt logs, learning.jsonl, false positives) is written under this single directory (see: lines 70-78, 703, 729, 748, 769, 1381, 2903, 3646, 3932, 4516).
- The `kilo.jsonc` configuration (in repo root) points to a venv python + absolute path to `mcp_server.py` — this lock-in works only on the machine that authored it.
- No `.env`, `BOUNTYBUD_ROOT`, or project-local discovery exists.
- Therefore every new machine / new IDE launch starts with an empty global `~/.bountybud`, causing:
  - "Fresh slate" learning DB
  - Lost target context
  - Inconsistent primitive/findings history
  - MCP tools (esp. `get_context`, `save_finding`, `log_hunt_event`) return empty results.

### Secondary Issues
- No `.bountybud/` in the repo root or per-CWD detection.
- No environment variable `BOUNTYBUD_ROOT` or config file (`bountybud.json`) for project-local override.
- `mcp_trigger.py` also reads global state indirectly via MCP endpoints.

**Conclusion:** The global `~/.bountybud` assumption is the single point of fragility that breaks continuity across machines and IDEs.

---

## Recommended Solution – Project-Local Data Directory

### Design Goals
1. **Zero-config portability** – running the MCP server from *any* directory automatically uses `./.bountybud`.
2. **Backward compatible** – existing `~/.bountybud` data is migrated once on first run if the local directory is empty.
3. **Explicit override** via env var or config file when needed.
4. **Fully self-contained** – all learning, state, and logs live inside the invocation root.
5. **IDE / CLI agnostic** – works the same whether launched from VS Code, Cursor, Claude Desktop, Claude Code, Windsurf, or raw `python mcp_server.py`.

### Core Change (in `mcp_server.py`)

Replace the static line 69:
```python
# NEW: Detect invocation directory and allow project-local storage
def _resolve_data_dir() -> str:
    # 1. Explicit env override (highest priority)
    if env := os.getenv("BOUNTYBUD_ROOT"):
        return os.path.abspath(env)

    # 2. Look for project config in CWD
    cwd = os.getcwd()
    config_candidates = [
        os.path.join(cwd, "bountybud.json"),
        os.path.join(cwd, ".bountybud", "config.json"),
        os.path.join(cwd, ".bountybudrc"),
    ]
    for cfg in config_candidates:
        if os.path.exists(cfg):
            try:
                with open(cfg) as f:
                    data = json.load(f)
                if "data_dir" in data:
                    return os.path.abspath(os.path.join(cwd, data["data_dir"]))
            except Exception:
                pass

    # 3. Default: create .bountybud inside current working directory
    local_dir = os.path.join(cwd, ".bountybud")
    return local_dir

DATA_DIR = _resolve_data_dir()
os.makedirs(DATA_DIR, exist_ok=True)
# (rest of sub-directory creation and LEARNING_DB line remain identical)
```

All subsequent hard-coded `DATA_DIR` join calls remain unchanged (they are already correct).

### Migration Logic (run once on first start)
```python
global_dir = os.path.expanduser("~/.bountybud")
if os.path.isdir(global_dir) and not os.listdir(DATA_DIR):
    # shallow copy of useful files
    for sub in ("targets", "primitives", "findings", "sessions", "huntlogs"):
        src = os.path.join(global_dir, sub)
        dst = os.path.join(DATA_DIR, sub)
        if os.path.isdir(src):
            shutil.copytree(src, dst, dirs_exist_ok=True)
    # also bring learning DB if it exists
    if os.path.exists(os.path.join(global_dir, "learning.jsonl")):
        shutil.copy2(os.path.join(global_dir, "learning.jsonl"),
                     os.path.join(DATA_DIR, "learning.jsonl"))
```

## Trade-offs & Clarification Questions

| Option | Pros | Cons |
|--------|------|------|
| **Local `.bountybud/` (recommended)** | True portability, works in every IDE/CLI, git-ignored state, simple mental model | Existing global `~/.bountybud` data must be migrated once |
| Keep global + env override | No migration needed | Loses one-command portability across machines/IDEs |
| Per-IDE config files | Some IDEs already do this | Fragile, different syntax per client, still global by default |

### Clarification on Launch Context (User Feedback)
User clarified that **BountyBud MCP server is invoked from arbitrary target project folders**, not from inside the `BountyBud/` repo directory.

- Therefore the `kilo.jsonc` (and any IDE `claude_desktop_config.json`, Cursor/Windsurf MCP settings, etc.) must contain **absolute paths** to the Python interpreter and `mcp_server.py`.
- However, the **data storage** logic in `mcp_server.py` must **still** resolve relative to `os.getcwd()` (the target project folder the user opened in the IDE). This enables per-project `.bountybud/` folders automatically.

**Result:** Two different concerns:
1. MCP *command registration* → absolute paths required.
2. Runtime data directory resolution → CWD-based (already implemented).

### Remaining Root Causes of Current IDE Failures (After Local-Data Change)
- **Hard import at top of `mcp_server.py:33`**:
  ```python
  from notifications import send_telegram_msg
  ```
  This pulls in `requests`, which may not exist in the venv used by the user's IDE. Even after installing `requests`, any IDE using a different Python or without it will fail on startup.
- `kilo.jsonc` currently (pre-fix) used an absolute path that only worked on the developer's laptop. Switching to relative paths (`./venv/...`) then broke users who launch from other directories.
- No helper or documented way to register the MCP server using the correct absolute paths for the user's local BountyBud installation.

## Updated Recommended Solution

### 1. Make `notifications` import lazy / optional (Critical Fix)
Change the top-level import to a lazy import inside the only function that uses it (`_send_telegram_alert` or equivalent around line 3557). This guarantees the MCP server starts successfully even when Telegram/requests is not installed.

```python
# Remove the top-level import
# from notifications import send_telegram_msg

# Replace the call site with:
def _safe_send_telegram(msg: str):
    try:
        from notifications import send_telegram_msg
        send_telegram_msg(msg)
    except Exception:
        logger.warning("Telegram notification skipped (requests or config missing)")
```

This eliminates the most common startup crash.

### 2. Restore / Correct `kilo.jsonc` to Use Absolute Paths
Keep (or generate) absolute paths in the example config. Example after installation:

```jsonc
{
  "mcp": {
    "bountybud": {
      "type": "local",
      "command": [
        "/absolute/path/to/BountyBud/venv/bin/python",
        "/absolute/path/to/BountyBud/mcp_server.py"
      ],
      "enabled": true
    }
  }
}
```

### 3. Provide Easy Registration Helper (Recommended)
Add a small convenience command, either:
- Update `bb-ctl.sh` with a `register-mcp` subcommand that prints the exact JSON snippet with the correct absolute paths for the current installation, or
- Create a one-line python helper `python -m bountybud.register_mcp`.

Example output users can copy-paste into their IDE settings.

## Trade-offs

- **Lazy import** (recommended): Zero user friction, graceful degradation. Minor extra code inside the one function.
- **Hard dependency**: Simpler code, but frequent startup failures across IDEs unless `requests` always present.
- **Absolute paths in registration**: Required because MCP clients spawn the server independently of CWD of the IDE window.
